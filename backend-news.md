■ 概要
- 平日毎朝、主要バックエンド技術・クラウド基盤・データベース・API開発ツールの公式リリースノート等から最新情報を集める
- 情報源は公式一次情報のみを使用する
- Notionのデータベース「バックエンド最新情報」にまとめる

■ 作業手順
1. 直近2〜3か月の「バックエンド最新情報」ページを確認する
2. 既出トピックを把握し、重複判定に使う情報を控える
3. 公式一次情報のみを確認し、候補トピックを作成する
4. 重複や軽微な更新を除外する
5. 「今日チームに共有する価値があるか」で3〜10件に絞る
6. Notionに「バックエンド最新情報 YYYY/MM/DD」ページを作成する
7. 日付プロパティをセットし、2セクション構成で本文を記載する

■ 出力フォーマット
- 3〜10トピックを選定
- 各トピックに必ず以下を含める：
  - 要約
  - 何が変わったか
  - 開発時の活用イメージ
  - 導入方法（最低でも「どこで有効化/設定するか」「まず何を試すか」がわかる粒度。
    分からない場合は「導入方法（要調査）」として不足情報を書く）
  - 注意点・制約
  - 公式リンク
  - 関連タグ（関連する言語、フレームワーク、データベース、クラウド、インフラ、カテゴリを1〜5個。例: Node.js, Go, PostgreSQL, Redis, AWS, Docker, セキュリティ）

■ 2セクション構成
  1) 今日の最新情報（その日の新規アップデート中心）
  2) 最近の重要アップデート（再掲）（直近2か月以内で、まだ試す価値が高いものを
     3件前後ランダムにピック。候補が無ければ「該当なし」と明記）

■ 重複排除
- 候補を作る前に、直近2〜3か月の「バックエンド最新情報」ページを確認し、既出の見出しは除外
  （重複判定はサービス名、機能名、公開日、公式URL、トピック見出しを基準にする）
- 一度「今日の最新情報」に投稿したトピックは、以後の「今日の最新情報」には再投稿しない
- 「今日の最新情報」と「再掲」で同じトピックを重複させない
- 同じ話題の続報は、前回からの差分が明確な場合のみ採用する

■ 情報ソース（優先順位）
- 公式一次情報のみを使用する
- 使用可能な情報源：
  - 公式Changelog
  - 公式Release notes
  - 公式Blog
  - 公式Docs
  - 公式GitHub Releases / Discussions / Issues（フレームワーク・ライブラリ・開発ツールの場合のみ）
- X、ニュース記事、個人ブログ、まとめ記事、解説記事は使用しない
- 公式リンクが見つからないトピックは採用しない
- 最優先: Node.js・Deno・Bun・PHP・Ruby・Python・Go・Java・.NET・PostgreSQL・MySQL・Redis・AWS・Google Cloud・Microsoft Azure
- 優先（重要度に応じ取捨選択）: Laravel・Ruby on Rails・Django・FastAPI・Spring Boot・NestJS・Hono・Prisma・Drizzle ORM・GraphQL・gRPC・Docker・Kubernetes
- 補助（話題性がある場合）: Supabase・Firebase・Cloudflare Workers / Pages・Vercel・Netlify・PlanetScale・Neon・MongoDB・Elasticsearch・OpenTelemetry・Terraform・GitHub Actions など
- 新しいバックエンド技術・クラウドサービス・データ基盤・API開発ツールが出た場合は、初回だけでもトピック化して共有する

■ フィルタリング基準
- 「今日チームに共有する価値があるか」で3〜10件に絞る。些細な修正は避ける
- 優先するもの：開発体験が変わる新機能 / ランタイム・フレームワークの破壊的変更 /
  セキュリティ・認証認可・コンプライアンス関連 / API設計・データベース・キュー・キャッシュへの影響 /
  パフォーマンス・スケーラビリティ・可用性への影響 / インフラ・CI/CD・監視・運用フローへの影響 /
  価格・制限・提供形態の変更
- 除外するもの：軽微なバグ修正 / 表記変更だけの更新 / 影響範囲が狭い内部改善 /
  一次情報で確認できない噂や予告

■ 投稿先と形式
- Notionのデータベース「バックエンド最新情報」に「バックエンド最新情報 YYYY/MM/DD」ページを作成し、
  2セクション構成で本文を記載（日付プロパティもセット）
- タグ/マルチセレクトのプロパティがある場合は、各トピックの関連タグを設定する

■ 不確実性の扱い
- 一次情報が見つからない/確信が持てない場合は採用しない
- 導入方法が不明な場合は、推測で書き切らず「要調査」として次に調べる観点を書く
- 「要調査」は公式一次情報に更新自体の根拠があるが、具体的な導入手順が不足している場合のみ使う
